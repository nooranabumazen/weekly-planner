# Weekly Planner PWA - Architecture Guide

## Overview
A personal weekly planner PWA built with React (Vite), Firebase (Auth + Firestore), deployed on Vercel. All styling is inline with CSS variables for theming. No external UI library.

## Data Architecture

### Firestore Structure
```
users/{uid}/
  weeks/{weekKey}          # e.g. "2026-03-30" (Monday date)
    tasks: { mon: [], tue: [], ..., sun: [], later: [] }
    _lastModified: timestamp
  
  meta/
    futureTasks       { items: [{ id, text, date }] }
    notebooks         { items: [{ id, title, content }] }
    journal           { entries: { "2026-03-30": "<p>HTML</p>", ... } }
    contacts          { items: [{ id, name, birthday, likes, dislikes, relationship, notes }] }
    archive           { items: [{ id, text, category, completedAt, assignedDate }] }  # Unlimited, never capped
    dailyHabits       { items: [...], _weekKey: "2026-03-30" }
    weeklyHabits      { items: [...], _weekKey: "2026-03-30" }
    settings          { categories, layout, notes, darkMode }
    habitHistory      { weeks: { "2026-03-23": { daily: [...], weekly: [...] }, ... } }  # Last 8 weeks
    recurringRules    { items: [{ type, text, category, day, count?, until?, pattern?, weekday?, nth?, dayOfMonth? }] }
  
  backups/{timestamp}      # Complete snapshots, max 7 kept
```

### Task Object
```js
{
  id: "t1711800000_a3f2",   // "t" + Date.now() + random suffix
  text: "Buy groceries",
  done: false,
  category: "cat_cooking",  // matches a category id
  recurring: {              // optional, present if task repeats
    type: "weeks",          // "weeks" | "until" | "monthly"
    count: 4,               // for type "weeks": how many more weeks
    day: "tue",             // which day column it belongs to
    // -- or for "until" --
    until: "2026-06-01",    // repeat until this date
    // -- or for "monthly" --
    pattern: "nth_weekday", // "nth_weekday" | "day_of_month" | "last_day"
    weekday: 3,             // 0=Sun..6=Sat (for nth_weekday)
    nth: 2,                 // 1st, 2nd, 3rd... (for nth_weekday)
    dayOfMonth: 15,         // (for day_of_month)
  }
}
```

### Week Key Format
Always the Monday date as ISO string: `"2026-03-30"`. Generated by `getWeekKey()` in usePlannerData.js.

## Critical Sync Architecture

**NO real-time listeners.** All `onSnapshot` listeners were removed after repeated data loss incidents. The app uses one-time reads (`getDoc`) only.

### How sync works:
1. **On app open:** Reads all data once from Firestore (10+ parallel reads). If ANY read returns `READ_ERROR`, shows error screen with Retry button. Never writes defaults over missing data.
2. **On task change:** Writes to Firestore with 250ms debounce. `latestTasksRef` ensures the latest state is always written (prevents stale closures).
3. **On page hide/close:** Flushes any pending debounced save immediately.
4. **On tab focus (visibility change):** If away > 5 seconds, flushes pending saves, waits 500ms, then reads latest from Firestore. If week changed during absence, full page reload.

### Data loss prevention:
- `READ_ERROR` sentinel (Symbol) distinguishes network errors from truly missing docs (null)
- Defaults only written for genuinely new users (ALL meta docs are null)
- Habit reset only runs when the habit doc actually exists with a different `_weekKey`
- Carry-forward aborts entirely if previous week read fails
- All task callbacks use `dataRef.current` (a ref) instead of closure variables

### Key pattern: `dataRef`
```js
const dataRef = useRef(data);
dataRef.current = data; // updated every render

// In callbacks:
const deleteTask = useCallback((col, id) => {
  const t = dataRef.current.tasks; // always fresh
  update({ tasks: { ...t, [col]: t[col].filter(x => x.id !== id) } });
}, []); // empty deps, never stale
```

## New Week / Carry-Forward Logic (usePlannerData.js)

When the app opens and the current week's doc doesn't exist:
1. Read previous week's doc
2. Carry forward all incomplete (not done) tasks to Monday
3. Generate recurring tasks from both:
   a. Inline recurring rules on last week's tasks (including done ones)
   b. Persistent `recurringRules` meta doc (survives task deletion)
4. Monthly rules: check each day of the new week against the pattern, place on matching day
5. Weekly rules: decrement count, remove rule on last repetition
6. Reset daily/weekly habits if `_weekKey` doesn't match current week
7. Save habit history snapshot before reset (last 8 weeks kept)
8. Write the new week doc

## Recurring Tasks

Rules are saved to the persistent `recurringRules` meta doc immediately when created (right-click > Repeat). This means they survive:
- Checking off the task as done
- Deleting the task
- Clearing the entire week

**Types:**
- `weeks`: repeat for N more weeks, count decrements each week, removed at 0
- `until`: repeat until a date, removed when past
- `monthly`: repeats forever. Patterns: `nth_weekday` (e.g. 3rd Wednesday), `day_of_month` (e.g. 15th), `last_day`

## Backup System

- **Schedule:** Twice daily around 12 AM and 12 PM. Checks every 5 minutes if in backup window. 10-hour throttle stored in `localStorage` (survives reloads).
- **Storage:** `users/{uid}/backups/{timestamp}`, max 7 kept, oldest auto-deleted.
- **Manual:** Export JSON button in Settings downloads complete backup. Restore from Settings > View Backups.

## Category System

- Categories stored in `settings.categories` array
- Each has: `{ id, name, color, keywords? }`
- Auto-detection: when typing a task, `autoDetectCategory()` matches against keywords
- Uses `\b` word boundary regex for single words (prevents "show" matching "shower")
- Keywords sorted longest-first before matching
- Multi-word keywords use `includes()` instead of regex

## Drag and Drop

- Uses HTML5 drag and drop API (not a library)
- `DragContext` (React context) tracks global drag state via `dragstart`/`dragend`/`drop` events
- `DropZone` components: 2px normally, expand to 8px during drag, 20px with gold dashed border on direct hover
- Auto-scroll when dragging near top/bottom edges (60px threshold)
- Drop state cleanup: capture-phase drop listener + 3-second dragover timeout fallback

## Search

- `getSearchResults()` searches across: tasks, archive, future tasks, notebooks, journal, contacts, quick notes, habits
- Each result has a `nav` object: `{ view, noteId?, date?, contactId? }`
- Clicking a result: switches view, navigates to the item, runs `highlightInDOM()`
- `highlightInDOM()`: DOM TreeWalker finds all text nodes matching the query, wraps first match in `<mark>` tag, scrolls to it
- For textareas (quick notes): selects matching text instead
- Highlights cleared on next click/keypress

## Firestore Security Rules
```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId}/{document=**} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
  }
}
```

## Important Conventions

- **No em dashes or en dashes** in user-facing text (owner preference)
- **No onSnapshot listeners** (data loss risk, use getDoc only)
- **All styles inline** (no CSS modules, no styled-components)
- **Fonts:** DM Sans (body), JetBrains Mono (labels/headers)
- **Archive is unlimited** (no cap on stored completed tasks)
- **Portrait lock** via PWA manifest + CSS media query
- **Spellcheck disabled** in tables only (via attribute on table elements)
- **Tab key** inserts tab space in editors, or indents/outdents lists
- **Dark mode text selection:** highlighted text auto-sets foreColor to dark

## Known Patterns to Preserve

1. `latestTasksRef` must always be updated before any write (prevents stale data)
2. `dataRef.current` in all task callbacks (prevents stale closures from React batching)
3. `READ_ERROR` sentinel must be checked before writing ANY data
4. Habit reset must check `_weekKey` matches, not just existence
5. Carry-forward must abort if previous week read fails
6. Recurring rules must be saved to persistent meta doc on creation, not just on the task
